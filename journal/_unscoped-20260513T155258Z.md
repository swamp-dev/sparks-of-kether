# Journal — unscoped: fix hand-magnify overflow + transform-order bug

Append-only. Never edit or delete past entries. One entry per `git push`
on this branch.

User-reported bug (no ticket): the #579 hand swell-to-centre magnify
was unusable — the whole hand container grew, a scrollbar appeared,
and the magnified card never reached viewport-centre.

---

## 2026-05-13T15:52:58Z — push 1: root-cause + fix

**Pushed:** fix(hand): magnify lift + overflow regressions from #579.

**Why:** two compounding CSS bugs in `components/hand/Hand.tsx`:

1. The inner `[data-hand-fan]` div carried `overflow-x-hidden`. Per
   the CSS overflow spec, when `overflow-x` is non-`visible` the
   browser implicitly coerces `overflow-y` from `visible` to `auto`.
   The magnified card's transform overflowed the fan vertically; that
   overflow then became scrollable content, the fan visibly grew, and
   the `absolute right-0 top-0` close-`×` rode up to the top of the
   scrollable area. Fix: swap to `overflow-x-clip` — clips the same
   way `hidden` does but does NOT establish a scroll container and
   does NOT trigger the implicit `overflow-y: auto`. Tailwind 3.4.19
   has the utility (added in 3.2).

2. The magnify transform was `${baseTransform} scale(3.5) translateY(-35vh)`.
   CSS applies transform functions right-to-left in the string, so the
   inner `translateY(-35vh)` ran in pre-scale local coords and then
   the outer `scale(3.5)` multiplied the result. Effective lift was
   3.5 × 35vh = 122.5vh — far above the viewport. The pre-fix comment
   misread this as "vh is unaffected by scale → translation is
   consistent across card sizes": the *unit* is unaffected but the
   composition multiplies the translation regardless of unit. Fix:
   reorder to `translateY(-35vh) scale(3.5)` so the translate is the
   last function applied, in post-scale (viewport) coords, lifting
   the intended 35vh.

The two bugs reinforced each other: bug 2 produced wild overflow; bug
1 turned that overflow into a scrollable region. Either alone would
have left the feature unusable; both had to be fixed.

**Notes:**

- The pre-fix #340 comment claimed `overflow-x-hidden` was load-bearing
  for selected-card stacking context. That claim was incorrect — `overflow`
  is not a stacking-context primitive in CSS. The actual stacking
  context the per-card `zIndex` participates in is provided by the
  outer `position: fixed` wrapper (and, when active, by the fan's own
  `transform`). Updated the comment to reflect this.
- E2E side-effect: `walkToPlayScreen` in `e2e/visual-regression.spec.ts`
  left the cursor at the Begin button's coordinates after
  `.click()`. At the mobile (375×667) viewport those coordinates land
  on a hand card after PlayScreen mounts, which fired the card's
  `mouseenter` and triggered the magnify. Pre-fix the magnified card
  was 122.5vh off-screen-up (invisible), so the screenshot still
  matched the rest-state baseline by accident. Post-fix the magnified
  card sits correctly at viewport centre — visible in the capture,
  which then diffed against the rest-state baseline. Parked the cursor
  at (0, 0) at the end of the walker so the capture is
  position-independent. The magnify behaviour is covered by the unit
  tests added in this push; a dedicated E2E screenshot of the
  magnified state is out of scope for this fix.
- Code-reviewer (one pass) flagged two significant comment-accuracy
  points: (a) the updated #340 comment implied `position: relative`
  alone was meaningful for z-ordering, when really the stacking
  context comes from the outer `position: fixed` (or the fan's
  transform when active), and (b) the right-to-left framing of the
  transform composition was phrased as "matrix level" instead of the
  simpler "right-to-left in the string". Both corrected. Also added a
  third regression assertion pinning that the base transform
  (`rotate(...) translateY(...)`) precedes the magnify pieces in the
  concat — so a future reverse of `baseTransform + magnifyTransform`
  wouldn't slip past.
